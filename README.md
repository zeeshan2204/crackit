## 📌 What is CrackIt?

CrackIt simulates real campus placement exams with 4 sections — **Numerical Ability, Verbal Ability, Reasoning, and Coding**. After the exam, it generates an **AI-powered performance analysis** with a personalized 7-day study plan using Groq's LLaMA3.

Built as a final year project with a production-grade tech stack — deployed live on Vercel + Railway + Neon.

---

## ✨ Features

### 🎯 Exam Engine
- **4-section placement exam** — Numerical · Verbal · Reasoning · Coding
- **500+ MCQ questions** across 3 sections (170 Numerical · 170 Verbal · 160 Reasoning)
- **22 coding problems** with live test case execution (Python, Java, C, C++)
- **10 random questions** picked per MCQ section every attempt — never the same exam twice
- **Per-section countdown timers** (placement exam style):
  - 🔢 Numerical Ability — 20 min
  - 📖 Verbal Ability — 15 min
  - 🧩 Reasoning Ability — 25 min
  - 💻 Coding — 45 min
- **Auto-submit** when section time runs out

### 📚 Question Coverage
- **Numerical**: Percentages, Profit & Loss, Time-Work, Speed-Distance, SI/CI, Pipes & Cisterns, Probability, Permutation & Combination, Mensuration, Logarithms, Data Interpretation (tables, pie charts, bar graphs)
- **Verbal**: Synonyms, Antonyms, Fill in the Blanks, Reading Comprehension (6 passages), Para Jumbles, Error Spotting, Active/Passive Voice, Direct/Indirect Speech, Cloze Test, One Word Substitution, Idioms & Phrases
- **Reasoning**: Number & Letter Series, Coding-Decoding, Blood Relations, Direction Sense, Syllogism, Analogies, Odd One Out, Seating Arrangement, Calendar, Clocks, Ranking, Venn Diagrams, Statement & Conclusion

### 🤖 AI Performance Analysis
- Personalized feedback generated by **Groq LLaMA3** after exam completion
- Covers: Overall assessment · Strengths · Focus areas · 7-day study plan
- Section-wise and difficulty-wise (Easy/Medium/Hard) performance breakdown
- Weak area action plans with section-specific preparation tips

### 📋 Answer Review
- Post-exam review of all MCQ sections with passage/table context shown for RC & DI questions
- ✅ Correct answers highlighted green
- ❌ Wrong answers highlighted red with correct answer shown
- ○ Skipped questions with correct answer revealed
- "Not attempted" message for skipped sections

### 💻 Live Code Execution
- Write and run code directly in the browser
- Executed on **deployed Railway backend** — works for all users, no localhost required
- Real test case validation with PASS/FAIL output per test case
- Supports **Python · Java · C · C++**

### 📊 Progress & Leaderboard
- Score trend chart (Recharts line graph) across multiple attempts
- Stats: Best score · Latest score · Average · Trend
- Section-wise score history table
- Global leaderboard ranking all users by total score

### 🔐 Authentication
- JWT-based auth with Access + Refresh token flow
- Email/Password registration and login
- **Demo Mode** — try the full exam without signing up (data resets on reload)

## 🚀 Setup & Installation

### Prerequisites
- Python 3.11+
- Node.js 18+
- PostgreSQL (or [Neon](https://neon.tech) free tier)
- Groq API key — free at [console.groq.com](https://console.groq.com)

### 1. Clone the repo
```bash
git clone https://github.com/zeeshan2204/ai-mock-interview.git
cd ai-mock-interview
```

### 2. Backend setup
```bash
cd backend

# Create virtual environment
python -m venv venv
venv\Scripts\activate          # Windows
source venv/bin/activate       # Mac/Linux

# Install dependencies
pip install -r requirements.txt

# Create .env file
cp .env.example .env
# Fill in your DATABASE_URL, JWT_SECRET, GROQ_API_KEY

# Start backend
uvicorn app.main:app --reload
```

### 3. Frontend setup
```bash
cd frontend
npm install

# Create .env file
echo "VITE_API_URL=http://localhost:8000/api" > .env

npm run dev
```

### 4. Open in browser
```
Frontend  →  http://localhost:5173
API Docs  →  http://localhost:8000/docs
```

## 📡 API Endpoints

```
POST /api/auth/register          Register new user
POST /api/auth/login             Login with email/password
POST /api/auth/refresh           Refresh access token
GET  /api/auth/me                Get current user

POST /api/exam/start             Start new exam session
GET  /api/exam/questions/{sec}   Get questions for section
POST /api/exam/submit-section    Submit section answers + score
POST /api/exam/complete/{id}     Complete exam + calculate total score
GET  /api/exam/result/{id}       Get exam result
GET  /api/exam/history           Get user's exam history
GET  /api/exam/leaderboard       Get global leaderboard

POST /api/ai/exam-feedback       Generate AI feedback (Groq LLaMA3)
POST /api/code/run               Execute code (Python/Java/C/C++)
```

Full interactive docs at `http://localhost:8000/docs`

## 💡 Key Technical Decisions

**Why 500+ questions in a JS file instead of the database?**
The question bank is frontend-only for fast, zero-latency random selection via `pickRandom(bank, 10)`. The backend DB stores session data and scores. Questions coming from a 500-question local bank ensures 10 unique questions per attempt with no DB round-trip — making the exam load instantly even on slow connections.

**Why sessionStorage for exam scores alongside the database?**
Dual-write strategy: scores are saved locally in sessionStorage AND sent to the backend. The result page reads from sessionStorage first (instant load), falling back to the API if needed. This makes the demo mode and result page bulletproof regardless of network conditions.

**Why per-section timers instead of one global timer?**
Real campus placement exams use strict per-section time limits. Implementing separate section timers (20/15/25/45 min) makes the exam experience authentically match what students will face in placement drives.

**Why Groq over OpenAI?**
Groq's free tier provides significantly faster inference with LLaMA3-8B (~500 tokens/sec). For a portfolio project, it offers the best cost-to-performance ratio without a paid API plan.

**Why Railway for code execution instead of Judge0?**
Judge0's free tier (50 req/day via RapidAPI) is too limited for an exam platform. Railway's backend runs code via Python subprocess with GCC and OpenJDK installed — giving unlimited execution with full control over the runtime environment.
